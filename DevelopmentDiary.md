## 1. 研究の位置づけと全体設計

### 1.1 背景と目的

本研究は、先輩が実装した **KHR-3HV の二足歩行 RL パイプライン**（Genesis + rsl-rl、
および実機動作）を土台に、**同じ機体で四足歩行を新規に獲得する**ことを目的とする。
二足ではなく四足を選ぶことで、より安定した歩行基盤の上で Sim2Real の成功率を検証する。

- ベースリポジトリ: `khr_rl_sample`（先輩由来、二足歩行）を fork した `khr_rl_tora`
- 二足版の中核 `khr_env.py` / `khr_train.py` は**触らず**、四足専用に
  `khr_quad_env*.py` / `khr_train_quad*.py` を新規作成して開発した。
- **ファイル分離の原則**: バージョンごとに env / train / record を複製し、番号を増やす。
  過去バージョンの学習結果 (`logs/`)・研究記録 (`experiments/`)・モデルには一切触れない。
  実機での検証がまだ済んでいないため、資産を完全に分けて追跡可能性を保つ方針。

### 1.2 技術スタック

| 要素 | 内容 |
|---|---|
| シミュレータ | Genesis (`genesis-world` 0.4.6, editable install) |
| RL ライブラリ | rsl-rl-lib ≥ 5.0（現 5.4.0）, PPO, `OnPolicyRunner` |
| DL フレームワーク | PyTorch 2.12.0, tensordict |
| 制御周期 | 50 Hz（`dt = 0.02`）。実機 Meridian は 100 Hz、ポリシーは 50 Hz |
| バックエンド | 学習・録画は GPU（`gs.gpu`）、評価ビューアは CPU |

### 1.3 四足形態の設計思想

二足ヒューマノイドを四足化するための機構的な工夫:

- **腕 = 前脚、脚 = 後脚**。URDF を `khr3hv_12dof.urdf`（腕 fixed）から
  `khr3hv.urdf`（腕 revolute）に変更し、肩・肘を可動にして前脚として使う。
- **胴体を y 軸まわりに +90° ピッチ**させ、「前のめりの四足姿勢」を初期姿勢とする。
  （初期の試作では −90°=後ろ倒れだったが、見た目・心理面から前倒れに変更。理由は §6.2）
- 制御対象は**全 22 関節**（腕 8 + 脚 12 + 胴/頭 yaw 2）。`num_actions = 22`。
- 接地点は 4 点: 前脚 `l_lowerarm` / `r_lowerarm`、後脚 `l_foot` / `r_foot`。
- 歩容は **4 脚トロット（対角ペア）**。対角の (FL,RR) と (FR,RL) を位相 0.5 ずらす。

## 2. システム・アーキテクチャ

### 2.1 ロコモーション座標系補正（本研究で最も重要な工夫の一つ）

胴体を 90° 倒しているため、**ベース座標系の各軸が世界座標とずれる**
（base-z = 世界前方、base-x = 世界下方）。速度追従・姿勢報酬をベース座標系のまま
計算すると「前進を罰する」等の座標系バグが起きる。これを防ぐため、報酬計算では
初期姿勢クォータニオンで世界と揃えた **loco frame** に変換する。

```python
self.loco_lin_vel = transform_by_quat(self.base_lin_vel, self.init_base_quat)
self.loco_ang_vel = transform_by_quat(self.base_ang_vel, self.init_base_quat)
projected_gravity_rel = transform_by_quat(self.projected_gravity, self.init_base_quat)
```

- 速度・コマンド・姿勢報酬 → **loco frame** で評価。
- 観測の `base_ang_vel` / `projected_gravity` → **base 座標系のまま**残す。
  （実機の IMU=ジャイロ/加速度計は機体座標で測るため、Sim と Real を揃える）
- この非対称が Sim2Real 整合の要。評価スクリプトで速度を測るときも `loco_*` を使う
  必要がある（base 座標のまま測ると旋回が −4% と誤って出る。§4.9 の実測時に遭遇）。

### 2.2 観測（`num_obs = 121`）

Actor が受け取る観測（`_update_observation`）:

| スライス | 内容 | 次元 | 座標系 |
|---|---|---|---|
| [0:3]    | base 角速度 | 3 | base |
| [3:6]    | 投影重力 (projected gravity) | 3 | base |
| [6:9]    | コマンド (vx, vy, wz) × スケール | 3 | loco |
| [9:31]   | 関節角の default からの偏差 | 22 | — |
| [31:53]  | 関節角速度 | 22 | — |
| [53:75]  | 現在の action | 22 | — |
| [75]     | cos(phase) | 1 | 歩容位相 |
| [76]     | sin(phase) | 1 | 歩容位相 |
| [77:99]  | 前ステップの action | 22 | — |
| [99:121] | 前ステップの関節角速度 | 22 | — |

### 2.3 特権観測（critic 用、`num_privileged = 80`）— 非対称 Actor-Critic

critic だけが観測できる真値（ノイズなし＋DR パラメータ）を与える。

`base_ang_vel(3) + projected_gravity(3) + dof_pos偏差(22) + dof_vel(22) + last_dof_vel(22)
+ loco_lin_vel(3) + 追加質量(1) + 摩擦(1) + COMシフト(3) = 80`

- `obs_groups`: actor → `policy`、critic → `policy` + `privileged`。
- loco 系の並進速度・DR パラメータは実機で観測不能なので actor には渡さない。

### 2.4 制御と歩容

- **PD 位置制御**: `kp = 25`, `kd = 0.5`, `action_scale = 0.15`。
  22 関節を `motors_dof_idx`（各関節の `dof_start`）で**インデックス指定バッチ制御**。
  DOF の内部順とコンフィグの関節順が一致しないため、名前→インデックスで明示的に対応付ける。
- **1 ステップの action 遅延**（`simulate_action_latency=True`）— 実機の遅延を模擬。
- **トロット位相**: `gait_period = 0.5 s`、`gait_offsets = [0.0, 0.5, 0.5, 0.0]`（FL,FR,RL,RR）。

### 2.5 報酬関数（`_reward_<name>` を自動探索）

| 報酬項 | 役割 |
|---|---|
| `tracking_lin_vel` | 指令並進速度への追従（**最重要**、v3 で 1.5→4.0） |
| `tracking_ang_vel` | 指令旋回速度への追従 |
| `orientation` | 胴体姿勢を公称に保つ |
| `base_height` | 目標高さの維持 |
| `lin_vel_z` / `ang_vel_xy` | 上下動・不要な回転の抑制 |
| `gait_contact` / `gait_swing` | トロット位相に合った接地/遊脚を促す |
| `contact_no_vel` | 接地脚の滑りを罰する |
| `feet_clearance` | 遊脚を目標高さまで持ち上げる（すり足抑制） |
| `hip_pos` | 脚の横開き（roll/yaw）を罰する。**v8 で hip_yaw/ankle_roll を追加** |
| `feet_orientation` | 足裏を水平に保つ（**v7 で新設**） |
| `alive` | 生存ボーナス |
| `action_rate` / `dof_vel` / `acceleration` / `joint_torques` | 動作の滑らかさ・省トルク |
| `collision` | 自己衝突の罰 |

報酬 scale は init 時に `dt` を一度だけ掛ける。scale と `_reward_` メソッドは名前で
1:1 対応が必須（片方だけあると壊れる）。

### 2.6 Sim2Real レシピ（ドメインランダム化・ノイズ）

先輩の実機二足で実証済みのレシピを v1 から継承:

- **ドメインランダム化**: 摩擦 [0.1,1.5]、追加質量 [−0.1,0.5]kg、COM シフト [−0.02,0.02]m、
  kp スケール [0.9,1.1]、kd スケール [0.8,1.2]。
- **観測ノイズ**: 角速度・重力・関節角・関節速度に加算（実機センサ誤差の模擬）。
- **1 ステップ action 遅延**（前述）。
- **push 外乱**: 学習中に胴体へランダム水平速度キックを与え転倒回復を学ぶ
  （v5 で有効化、v6 でカリキュラム化）。

### 2.7 URDF・関節の物理制約（設計を縛る重要事実）

- **サーボ定格トルク 1.373 Nm**（実機 KONDO サーボ）を URDF の effort limit に設定。
  Genesis がこれを強制するため、飽和すると Sim でも指令通り動けない＝実機で発熱・脱調。
- **膝は片方向のみ**: `knee_pitch` の可動域は **[0.000, 2.793] rad**（負に曲がらない）。
  これが v8 で判明する「膝ロック」問題の物理的根源（§4.10）。
- 足裏水平の幾何条件: `hip_pitch + knee_pitch + ankle_pitch = −θ_base = −1.5708`。
  初期姿勢を作り直すときは常にこの条件を満たす。

## 3. 実行方法（コマンド集）

すべて venv 有効化後、`khr_rl_tora/` 直下から実行（URDF 参照が相対パスのため）。
GPU が必要（学習・録画）。録画は Chrome Remote Desktop/VS Code Remote 環境で
ビューアが見えないため、オフスクリーンで MP4 に保存する方式を採る。

```bash
source /home/tora/gsw4/bin/activate
cd /home/tora/gsw4/khr_rl_tora

# 学習（例: v8）。logs/khr-quadruped8/ に checkpoint と TensorBoard を出力
python khr_train_quad8.py -e khr-quadruped8 -B 4096 -I 4000 --seed 1

# 評価（ビューア。CPU）
python khr_quad_eval.py -e khr-quadruped8 --ckpt 3999

# 録画（ヘッドレス、MP4）。録画には X ディスプレイの指定が要る
export DISPLAY=:1
python khr_quad_record5.py -e khr-quadruped8 --ckpt 3999 --vx 0.4 --seconds 10 -o walk_v8_forward.mp4
#   方向指定: --vx 前進 / --vy 横 / --wz 旋回（未指定成分は 0）

# 研究記録の自動生成（report.md + metrics.csv を experiments/<exp>/ に出力）
python khr_quad_report.py -e khr-quadruped8

# 学習曲線
tensorboard --logdir logs
```

- **注意**: `khr_train_quad*.py` は同名 `exp_name` の `logs/<exp>/` を実行開始時に削除して
  作り直す。残したい結果は別 exp 名にするか事前に退避する。
- **録画スクリプトの世代対応**: `khr_quad_record.py` は v1 env を import する。v8 は
  `khr_quad_record5.py`（`khr_quad_env5` を読む複製）を使う。
- **`export DISPLAY=:1` が必須**: シェルで `DISPLAY` 未設定でも `/tmp/.X11-unix/X1` が
  存在するため、これを指定しないと録画が "No display" で失敗する。

## 4. バージョン別 開発日記（v1〜v8）

各バージョンは env / train を複製して番号を増やし、直前版からの差分のみを変える
（ablation を明確化するため）。共通の固定値: `num_envs=4096`, `seed=1`,
PPO `[128,64,32]` ELU, `kp=25/kd=0.5`, `action_scale=0.15`, `init_std=0.5`,
`entropy_coef=0.01`, `lr=1e-3`, `gamma=0.99/lam=0.95`。

### 4.1 v1 — 初の四足歩行成功（`khr-quadruped`, 3000 iter）

- **動機**: 二足版 `khr_env.py` を四足化し、まず「転ばずに前進する」ことを成立させる。
- **主な実装**: `khr_quad_env.py` を新規構築（22DOF、トロット、loco frame 補正、121 次元観測、
  非対称 actor-critic）。コマンドは前進のみ `vx ∈ [0.0, 0.2]`。
- **初期のつまずきと修正**: 最初の run はエピソード長 12.5 のまま伸びず**完全失敗**。
  探索 std が早期崩壊していたため、`init_std 1.0→0.5`, `action_scale→0.15`,
  `entropy_coef→0.01` に緩めて**探索崩壊を回避**（commit `e8d346b`）。
- **結果**: エピソード長 12.5 → 1001（転ばず立てる）。前進追従報酬 最終 1.45。
  ただし **実測速度 約 0.036 m/s vs 指令 0.15**＝**すり足で非常に遅い**。
- **分析 / 次への課題**: 歩けたが遅い。速さと自然さの改善が必要。

### 4.2 v2 — 「速く・自然に」の試み（設定 3 箇所のみ、失敗から学ぶ）

- **動機**: 新報酬を足さず既存パラメータ調整だけで v1 の遅さ・すり足を改善。
- **変更点**: `tracking_sigma 0.25→0.15`（追従を厳しく）、`vx [0,0.2]→[0,0.4]`、
  `feet_height_target 0.035→0.06`（遊脚を高く）。
- **結果**: **かえって遅くなった**（前進追従 最終 1.12、v1 の 1.45 より低下）。
- **分析（重要）**: 追従判定を厳しくしても速くならなかった。報酬内訳を精査すると
  **静止でも前進追従報酬の約 94% が得られる**一方、動くと罰（acceleration /
  similar_to_default / action_rate）が増え「**歩く報酬 < 歩く罰**」となり、
  静止が最適解になっていた（探索は潰れていない＝報酬設計の問題）。
- **次への課題**: 報酬「バランス」の是正。厳しさではなく割の合う設計に。

### 4.3 v3 — 報酬バランス是正（ブレークスルー）

- **動機**: v2 の診断に基づき「歩くこと」を明確に割に合わせる（設定 4 箇所のみ）。
- **変更点**: `tracking_lin_vel 1.5→4.0`、`similar_to_default −0.1→−0.02`、
  `action_rate −0.05→−0.02`、`acceleration −1e-4→−2e-5`（実効最大の罰を 1/5）。
  sigma=0.15 / command[0,0.4] / feet_height=0.06 は v2 を維持。
- **結果**: **前進追従報酬 最終 3.89（v1 の 2.7 倍）**。指令 0.3 に対し実速度 0.257 m/s
  で**追従 86%**。「実際に速く歩く」四足が完成。
- **先行研究の裏付け**: Rudin et al. 2021 (legged_gym) の ablation で速度追従報酬が
  最重要と示されており、`tracking_lin_vel` の増強が効いたことと整合。
- **分析 / 次への課題**: 前進は解決。次は「コマンドで自在に歩ける」全方向化へ。

### 4.4 v4 (1a) — 全方向歩行への拡張

- **動機**: Sim2Real に向け、まず横移動・旋回を含む「自在に歩ける四足」に仕上げる。
- **変更点**: `command_cfg` のみ。`vy [0,0]→[−0.15,0.15]`、`wz [0,0]→[−0.5,0.5]`。
  `vx [0,0.4]` は据え置き（変更を「横・旋回の追加」だけに限定し ablation を明確化）。
  報酬・環境は v3 のまま（環境は既に x/y と yaw を扱えるためコマンド範囲を開くだけで学習可）。
- **結果**: 全方向歩行を獲得。前進追従報酬 最終 3.91。前進追従 ≈86%、**横 ≈77%**。
  足裏の傾き 12.4°、トルク定格 95%超に張り付く関節 2/22。**以降の比較基準（ベースライン）**。
- **分析 / 次への課題**: 押し外乱に対する頑健性は横 −y が最弱（0.8 で耐え 1.0 で全滅）。
  実機に向けて外乱耐性を上げたい（→ push 外乱の導入）。

### 4.5 v5 (1b) — push 外乱で頑健化（頑健性↔性能のトレードオフを観測）

- **動機**: 先輩 config に枠だけあって未実装だった push を有効化し、実機の外乱耐性を上げる。
- **実装**: 環境を **`khr_quad_env2.py`** に変更し `_push_robots()` を追加。
  一定間隔で胴体の水平速度にランダムキックを加える。
  `push_interval_s 5→3`、`max_push_vel_xy 0.2→1.0`、push モードを有効化。
- **結果**: 前方・横 +y の押し耐性を +0.5 m/s 改善。しかし**歩行追従が低下**
  （前進 93→69%、横 78→35%、旋回 87→64%）、後方耐性はやや悪化、最弱の横 −y は不変。
- **分析**: **最初から最大 1.0 m/s の強い外乱**を与えたため方策が「守り」に入り歩行が鈍った。
  典型的な**頑健性↔性能のトレードオフ**。
- **次への課題**: 外乱を徐々に強めるカリキュラムで両立を狙う。

### 4.6 v6 (1b) — push カリキュラム（頑健性と性能の両立）

- **動機**: v5 のトレードオフを、外乱をカリキュラムで漸増させて解消する。
- **実装**: 環境を **`khr_quad_env3.py`** に変更。`push_curriculum_steps=50000` を追加し、
  `push_scale = min(1.0, step_counter / curriculum_steps)` で押し強度を 0→max に線形増加。
  まず歩行を習得させ、その後で押しを強める。押し間隔・最大強度 (3s, 1.0m/s) は v5 のまま。
- **結果**: 前進追従報酬 最終 3.49。歩行性能を保ちつつ外乱耐性を回復（v5 の劣化を解消）。
- **分析 / 次への課題**: 頑健性は改善したが、v4→v6 の過程で**足の動きが不自然**に。
  「足の縁で歩く」挙動が悪化していることが実測で判明。

### 4.7 v7 — 足裏の水平接地（ベタ足歩行）

- **動機**: v4→v6 で足裏が傾き「縁で歩く」挙動が悪化していた。実測で
  足裏傾き v4 12.4° → v5 22.7° → v6 24.3°、`ankle_pitch` −6.3→−34.4°、
  `ankle_roll` −12.6→−29.7°、`hip_yaw` +6.0→+20.0°（脚が外にねじれる）。
  これは push に耐える「踏ん張り姿勢」に流れた副作用で、`ankle_roll`/`hip_yaw` が
  **定格トルク 1.373Nm を連続使用**する原因＝実機の発熱・脱調リスクでもあった。
- **実装**: 環境を **`khr_quad_env4.py`** に変更し `_reward_feet_orientation` を新設
  （足裏法線の水平成分を罰する＝水平なら 0）。`reward_scales` に
  `"feet_orientation": -1.0` を追加。push カリキュラムは v6 のまま維持。
- **結果**: 足裏傾きが大幅改善（≈11.0°）。前進追従報酬 最終 3.53。
- **分析（重要な副作用の発見）**: 足裏は水平化したが、**定格 95%超に張り付く関節が
  7/22 に増加**。`hip_yaw +20.0°`、`ankle_roll −29.7°` と脚が大きく外に開いたまま保持され、
  それらが時間の 98〜99% トルク飽和していた。さらに詳細計測で、後脚の主動力である
  **膝がほとんど動いていない**ことが判明（次で解明）。
- **次への課題**: 「関節が動かない・ぎこちない」の根本原因を突き止め、トルク飽和を下げる。

### 4.8 v8 — 膝の解放と脚の横開き抑制（最新・Sim2Real 最有力）

- **動機**: v7 の診断で 2 つの根本問題が判明。
  1. **膝ロック**: `knee_pitch` の初期値 0.10 rad は機械的下限 0.0 の**わずか 5.7° 上**。
     膝は片方向にしか曲がらないため、主動力の膝がハードストップに押し付けられ、
     **可動幅わずか 6.4° なのにトルク張り付き 100%**。これが「関節が動かない・
     ぎこちない」の正体だった。
  2. **脚の横開き**: `hip_yaw ±20°`, `ankle_roll ±30°` で開いたまま保持され、その 4 関節が
     定格 95%超に時間の 98〜99% 張り付き（発熱・脱調リスク）。
- **実装**:
  - **初期姿勢の再導出**（足裏水平条件 hip+knee+ankle=−1.5708 を維持したまま膝に可動余地）:
    `knee_pitch 0.10→0.40`（下限までの余裕 5.7°→22.9°）、`hip_pitch −1.671→−1.770`、
    `ankle_pitch 0.0→−0.201`、`shoulder_pitch −1.5708→−1.771`、`elbow_pitch 0.0→0.20`
    （後脚が短くなる分、前脚も曲げてリーチを一致。前後差 3.0mm→2.0mm）、
    `base_init_pos z 0.197→0.1946`、`base_height_target` も同値。
  - 環境を **`khr_quad_env5.py`** に変更し、`_reward_hip_pos` の罰対象に
    `hip_yaw` / `ankle_roll` を追加（横開きをさらに抑制）。
  - **静的検証**: 新姿勢は 400 step で base_z 0.193 維持・x ずれ 1.7mm と安定。
    静止保持時のトルク定格 95%超は 0/22（膝はわずか 7〜8%）。
- **結果（v7 → v8）**:

  | 指標 | v7 | v8 |
  |---|---|---|
  | 膝の可動幅 | 6.4° | **22.3°**（3.5 倍） |
  | 膝のトルク張り付き | 100% | **3%** |
  | 定格 95%超に張り付く関節数 | 7/22 | **2/22**（v4 水準まで回復） |
  | `hip_yaw` のズレ | +20.0° | **+1.7°** |
  | `ankle_roll` のズレ | −29.7° | **−5.3°** |
  | 足裏の傾き | ≈11.0° | **7.5°（全版で最良）** |
  | 前進追従 | 95% | 92% |
  | 横追従 | 58% | 62% |
  | 旋回追従 | 85% | 91% |
  | 後方 push 耐性 | 1.5 | **2.5+ m/s** |
  | 横 push 耐性 (+y/−y) | 1.5 | 1.2 m/s |

- **分析**: 主目的（膝の解放・トルク飽和の低減）を達成。実機で最も効くのは
  **トルク飽和の解消（7/22→2/22）**で、発熱・脱調という実害を直接下げる。動作の自然さ・
  足裏接地・旋回性能も最良。**トレードオフ**として横 push 耐性が 1.5→1.2 に低下したが、
  これは v7 の「脚を大きく開いて踏ん張る姿勢」（＝トルク飽和の原因そのもの）を
  正常化した代償であり想定内。
- **総合評価**: **v8 が現時点で Sim2Real 用の最良モデル**。

### 4.9 v8 の 6 方向動作記録（前後・左右・左右旋回）

## 6. 重要な失敗と学び（トラブルシューティング）

研究上、失敗の記録は再現防止と考察の一次資料になるため明示的に残す。

### 6.1 探索の早期崩壊（v1）
最初の run はエピソード長 12.5 のまま学習が崩壊。`init_std` が高すぎて探索 std が
早期に潰れていた。`init_std 1.0→0.5`, `action_scale→0.15`, `entropy_coef→0.01` で回避。
**教訓**: 「学習が全く進まない」ときはまず探索量（std）の推移を疑う。

### 6.2 後ろ倒れ四足になった理由（初期姿勢）
最初の試作は胴体ピッチ −90°＝**後ろに倒れる**四足だった。KHR には首（ピッチ）関節が
無く、頭を前に向けられないため、頭を上に保つと自然に後傾解になっていた。見た目・心理面
（「後ろ倒れはホラー」）から**前倒れ（+90°）に変更**。機能上の問題ではなく設計判断。

### 6.3 「速くしようとして遅くなる」報酬設計の罠（v2→v3）
追従判定を厳しくする（sigma を下げる）だけでは速くならない。**静止でも追従報酬の大半が
得られ、動くと罰が勝つ**なら静止が最適解になる。解決は「厳しさ」ではなく
「歩く報酬 > 歩く罰」に**バランスを組み替える**こと（tracking を上げ、動作系の罰を下げる）。

### 6.4 頑健性↔性能のトレードオフ（v5→v6）
強い外乱を最初から与えると方策が守りに入り歩行が鈍る。**カリキュラム**（まず歩行、
その後で外乱漸増）で両立。legged_gym の定番と整合。

### 6.5 報酬の副作用の連鎖（v6→v7→v8）
足裏を水平化する報酬（v7）を足したら、別の関節（hip_yaw/ankle_roll）が飽和した。
さらに調べると膝が下限にロックしていた（v8 で解明）。**一つの報酬追加が別の関節に
しわ寄せする**ため、単一指標でなく「全 22 関節のトルク duty・可動幅」を横断で監視する
必要がある。これが v8 の詳細計測（膝 ROM・duty）につながった。

### 6.6 膝の片方向可動域の見落とし（v8）
`knee_pitch` の可動域 [0.0, 2.793] は片方向。初期値 0.10 は下限の 5.7° 上に過ぎず、
主動力の膝が事実上ロックしていた。**URDF の joint limit を初期姿勢設計に織り込む**べきだった。

### 6.7 計測時の座標系ミス（v8 6 方向計測）
旋回速度を base 座標系で測ると −4% と誤って出た。胴体 90° ピッチのため
**`loco_ang_vel` を使う**必要がある（§2.1）。報酬計算だけでなく評価計測でも同じ補正が要る。

### 6.8 GPU の競合（運用上の教訓）
録画（GPU）を学習中に実行し、学習を落としたことがある。**学習中に GPU 作業を重ねない**。

### 6.9 実機ネットワークの誤診（記録）
UDP タイムアウトの原因を最初 ufw と推測したが、`ufw status` は非アクティブ＝遮断していない
（無実）。真因は **PC の IP が DHCP なのに ESP32 の `keys.h` が送信先 IP を固定**していたこと。
実機フェーズで PC の IP 固定（DHCP 予約）が必要。

## 7. 残課題と今後の方針

### 7.1 シミュレーション内の残課題
- **横移動の左右非対称**（左 63% / 右 92%）: 最大の改善余地。URDF 非対称か報酬の
  左右不均等が疑い。
- **横 push 耐性の回復**（1.2→1.5）: v8 で正常化と引き換えに低下。踏ん張りに頼らない
  形での回復が課題。
- **後退の未対応**: 必要なら `lin_vel_x_range` を負に拡張して再学習。

### 7.2 Sim2Real（最終目標）に向けて
- 実機ファイル（`Meridian_consoleS_quad.py` / `khr_quad_env_no_gs.py`）はまだ
  バージョン管理外。fork 元 `s-kajita/Meridian_consoleS` には push しないため、
  管理方針（`khr_rl_tora` 配下へ移す等）の決定が必要。
- **符号キャリブレーション**（`genesis2khr_dir` が全 +1 で未校正）、
  **腕の荷重負担検証**、**PC の静的 IP 設定**が実機動作の前提ブロッカー。
- サーボ対応: L 系統 Meridim index = 21 + ID×2、R 系統 = 51 + ID×2。ポリシー 50Hz /
  Meridim 100Hz。
